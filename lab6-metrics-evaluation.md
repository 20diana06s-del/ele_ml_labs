## Т1: Почему метрики важны

**Метрика** — это численный показатель, который говорит нам, насколько хорошо модель решает задачу.

Без метрик невозможно:
- Сравнить две модели между собой
- Понять, улучшилась ли модель после изменений
- Решить, готова ли модель к использованию в продакшене

### Связь метрики с бизнес-задачей

Метрика должна отражать реальную цель. Если цель бизнеса — минимизировать пропуски мошеннических транзакций, метрика должна штрафовать за такие пропуски.

**Пример:** Банк хочет находить мошеннические транзакции. Можно оптимизировать accuracy, а можно — recall (полноту) для класса «мошенничество».

### Пример: Accuracy 99% может быть бесполезной

Представьте датасет с 10 000 транзакций:
- 9 900 нормальных (99%)
- 100 мошеннических (1%)

Модель, которая всегда предсказывает «нормальная транзакция», имеет accuracy 99%! Но она не нашла ни одного мошенничества.

**Вывод:** Высокая метрика не всегда означает хорошую модель. Важно выбирать метрику, соответствующую задаче.

## Т2: Метрики классификации (базовые)

### Confusion Matrix (Матрица ошибок)

Основа всех метрик классификации — матрица ошибок (confusion matrix).

Для бинарной классификации (положительный и отрицательный класс):

| | Предсказано: Положительно | Предсказано: Отрицательно |
|---|---|---|
| **Реально: Положительно** | TP (True Positive) | FN (False Negative) |
| **Реально: Отрицательно** | FP (False Positive) | TN (True Negative) |

**Обозначения:**
- **TP (True Positive)** — правильно предсказали положительный класс
- **TN (True Negative)** — правильно предсказали отрицательный класс
- **FP (False Positive)** — ошибочно предсказали положительный (ошибка I рода)
- **FN (False Negative)** — ошибочно предсказали отрицательный (ошибка II рода)

### Accuracy (Точность)

$$\text{Accuracy} = \frac{TP + TN}{TP + TN + FP + FN}$$

**Что показывает:** Долю правильных предсказаний среди всех.

**Когда работает:** Когда классы сбалансированы и ошибки обоих типов равноценны.

**Когда НЕ работает:** При сильном дисбалансе классов (см. Т3).

### Precision (Точность)

$$\text{Precision} = \frac{TP}{TP + FP}$$

**Что показывает:** Долю действительно положительных среди предсказанных положительных.

**Вопрос на который отвечает:** «Из всех, кого мы назвали положительными, сколько действительно положительные?»

**Когда важна:** Когда опасны ложные срабатывания (FP).
- Спам-фильтр: не важно пропустить спам, но плохо отправить важное письмо в спам
- Рекомендации: не важно не порекомендовать что-то, но плохо порекомендовать неинтересное

### Recall (Полнота)

$$\text{Recall} = \frac{TP}{TP + FN}$$

**Что показывает:** Долю найденных положительных среди всех положительных.

**Вопрос на который отвечает:** «Из всех действительно положительных, сколько мы нашли?»

**Когда важен:** Когда опасны пропуски (FN).
- Диагностика болезней: пропустить болезнь опаснее, чем сделать лишний анализ
- Мошенничество: пропустить мошенничество опаснее, чем проверить лишнюю транзакцию

### F1-score

$$F_1 = 2 \cdot \frac{\text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}}$$

**Что показывает:** Гармоническое среднее между Precision и Recall.

**Зачем нужна:** Когда нужно одновременно контролировать и Precision, и Recall. Гармоническое среднее «наказывает» за крайние значения (если одна метрика 0, F1 = 0).

### Трейдофф Precision vs Recall

Precision и Recall обычно находятся в обратной зависимости:
- Повышаем порог → меньше FP → выше Precision, но ниже Recall
- Понижаем порог → меньше FN → выше Recall, но ниже Precision

**Аналогия с рыбалкой:**
- Precision: какая доля пойманного — это рыба (а не мусор)
- Recall: какую долю всей рыбы в озере мы поймали

Можно поймать всю рыбу, но с большим количеством мусора (высокий Recall, низкий Precision). Можно ловить аккуратно, но пропустить много рыбы (высокий Precision, низкий Recall).

## Т3: Проблема несбалансированных классов

### Почему accuracy подводит при дисбалансе

**Пример:** Детекция мошеннических транзакций
- 99% транзакций нормальные
- 1% мошеннические

Модель «всегда нормальная» имеет accuracy 99%, но не находит ни одного мошенничества. Это бесполезная модель, но accuracy высокая!

### Примеры дисбаланса в реальности

| Область | Положительный класс | Дисбаланс |
|---------|---------------------|-----------|
| Банковское мошенничество | Мошеннические транзакции | 0.1-1% |
| Медицина | Редкие заболевания | 0.01-1% |
| Производство | Брак | 1-5% |
| Поиск | Релевантные документы | <0.1% |

### Метрики для дисбаланса

**Balanced Accuracy** — среднее accuracy по классам:

$$\text{Balanced Accuracy} = \frac{\text{Recall}_+ + \text{Recall}_-}{2}$$

**F1-score (weighted, macro):**
- **Macro F1:** среднее F1 по всем классам (все классы равны)
- **Weighted F1:** взвешенное по размеру классов (учитывает дисбаланс)

**ROC-AUC:** площадь под ROC-кривой. Может быть оптимистичной при сильном дисбалансе — когда отрицательный класс сильно преобладает, FPR может быть обманчиво низким.

## Т4: Метрики регрессии

### MAE (Mean Absolute Error)

$$\text{MAE} = \frac{1}{n} \sum_{i=1}^{n} |y_i - \hat{y}_i|$$

**Что показывает:** Среднюю абсолютную ошибку в тех же единицах, что и целевая переменная.

**Преимущества:**
- Интерпретируема (средняя ошибка в рублях/метрах/etc.)
- Устойчива к выбросам (робастна)

**Недостатки:**
- Не учитывает направление ошибки

### MSE (Mean Squared Error)

$$\text{MSE} = \frac{1}{n} \sum_{i=1}^{n} (y_i - \hat{y}_i)^2$$

**Что показывает:** Среднеквадратичную ошибку.

**Преимущества:**
- Математически удобна (дифференцируема)
- Штрафует большие ошибки сильнее

**Недостатки:**
- Единицы измерения — квадрат целевой переменной
- Чувствительна к выбросам

### RMSE (Root Mean Squared Error)

$$\text{RMSE} = \sqrt{\text{MSE}} = \sqrt{\frac{1}{n} \sum_{i=1}^{n} (y_i - \hat{y}_i)^2}$$

**Что показывает:** Корень из MSE, возвращает единицы измерения целевой переменной.

**Когда использовать:** Когда нужно наказывать за большие ошибки, но в понятных единицах.

### R² (Коэффициент детерминации)

$$R^2 = 1 - \frac{\sum_{i=1}^{n} (y_i - \hat{y}_i)^2}{\sum_{i=1}^{n} (y_i - \bar{y})^2}$$

**Что показывает:** Долю дисперсии целевой переменной, объяснённую моделью.

**Интерпретация:**
- R² = 1: идеальное предсказание
- R² = 0: модель не лучше предсказания средним
- R² < 0: модель хуже предсказания средним (очень плохо)

**Пример:** R² = 0.85 означает, что модель объясняет 85% дисперсии целевой переменной.

### MAPE (Mean Absolute Percentage Error)

$$\text{MAPE} = \frac{100\%}{n} \sum_{i=1}^{n} \left| \frac{y_i - \hat{y}_i}{y_i} \right|$$

**Что показывает:** Среднюю процентную ошибку.

**Преимущества:**
- Интерпретируема в процентах
- Позволяет сравнивать модели для разных масштабов

**Недостатки:**
- Не определена при y = 0
- Асимметрична (штрафует за предсказание выше реального сильнее)

### Сравнение метрик регрессии

| Метрика | Единицы | Чувствительность к выбросам | Когда использовать |
|---------|---------|----------------------------|-------------------|
| MAE | Оригинальные | Низкая | Когда выбросы не важны |
| MSE | Квадрат оригинальных | Высокая | Когда большие ошибки критичны |
| RMSE | Оригинальные | Средняя | Баланс между MAE и MSE |
| R² | Безразмерная (0-1) | Средняя | Для интерпретации качества |
| MAPE | Проценты | Средняя | Для сравнения разных масштабов |

## Т5: Валидация моделей

### Почему нельзя оценивать на train

Если оценить модель на тех же данных, на которых она обучалась, получим оптимистичную оценку. Модель «запомнила» обучающую выборку, но может плохо работать на новых данных.

**Аналогия:** Студент запомнил ответы на учебные задачи, но не понял принцип. На экзамене с новыми задачами провалится.

### Train/Test Split

Разделяем данные на две части:
- **Train (обучающая выборка):** для обучения модели (обычно 70-80%)
- **Test (тестовая выборка):** для оценки качества (20-30%)

```
|--------- Train ---------|-- Test --|
```

**Правило:** Тестовая выборка используется ТОЛЬКО один раз — для финальной оценки.

### Кросс-валидация (k-fold Cross-Validation)

Делим данные на k частей (фолдов). k раз обучаем модель на k-1 части и оцениваем на оставшейся.

**Пример для k=5:**

```
Итерация 1: | Test | Train | Train | Train | Train |
Итерация 2: | Train | Test | Train | Train | Train |
Итерация 3: | Train | Train | Test | Train | Train |
Итерация 4: | Train | Train | Train | Test | Train |
Итерация 5: | Train | Train | Train | Train | Test |
```

**Результат:** k оценок, берём среднее.

**Преимущества:**
- Более надёжная оценка
- Используем все данные для оценки

**Типичные значения k:** 5, 10

### Stratified Split

При разделении данных важно сохранить пропорции классов. Stratified split гарантирует, что в train и test будет одинаковая доля каждого класса.

**Зачем нужно:** Если в данных 5% класса «1», то и в train, и в test должно быть ~5% класса «1».

### Когда что использовать

| Ситуация | Метод |
|----------|-------|
| Много данных (>100 000) | Train/Test split |
| Мало данных (<10 000) | k-fold CV |
| Несбалансированные классы | Stratified k-fold CV |
| Финальная оценка | Отложенная выборка (holdout) |

## Т6: Threshold selection

### Что такое порог классификации

Многие модели классификации выдают вероятность (или скор) принадлежности к классу. Порог (threshold) — значение, выше которого мы считаем предсказание положительным.

**По умолчанию:** порог = 0.5

**Пример:**
- Модель выдала вероятность мошенничества = 0.7
- При пороге 0.5: транзакция блокируется (0.7 > 0.5)
- При пороге 0.8: транзакция пропускается (0.7 < 0.8)

### Как выбор порога влияет на Precision/Recall

| Порог | Precision | Recall | Интерпретация |
|-------|-----------|--------|---------------|
| Высокий (0.8-0.9) | Высокий | Низкий | Консервативно: мало FP, много FN |
| Низкий (0.1-0.3) | Низкий | Высокий | Агрессивно: мало FN, много FP |
| Средний (0.5) | Баланс | Баланс | Золотая середина |

### Выбор порога под бизнес-задачу

**Пример 1: Обнаружение мошенничества**
- Пропуск мошенничества (FN) стоит $1000
- Ложная блокировка (FP) стоит $10 (недовольный клиент)
- **Решение:** Низкий порог (агрессивно), максимизировать Recall

**Пример 2: Спам-фильтр**
- Пропуск спама (FN) не критичен
- Отправка важного письма в спам (FP) — потеря бизнеса
- **Решение:** Высокий порог (консервативно), максимизировать Precision

**Пример 3: Медицинская диагностика**
- Пропуск болезни (FN) может стоить жизни
- Ложная тревога (FP) — лишний анализ
- **Решение:** Очень низкий порог, Recall > 95%

### Как выбрать оптимальный порог

1. Определить стоимость FP и FN
2. Протестировать модель с разными порогами
3. Выбрать порог с учётом стоимости ошибок
4. Проверить на отложенной выборке

## Т7: Cost-sensitive learning

### Разная стоимость ошибок

В реальности ошибки FP и FN обычно имеют разную стоимость.

**Примеры:**

| Область | FP (ложная тревога) | FN (пропуск) | Что дороже? |
|---------|---------------------|--------------|-------------|
| Мошенничество | Проверка транзакции | Потеря денег | FN |
| Спам | Потеря письма | Несколький секунд чтения спама | FP |
| Медицина | Лишний анализ | Пропуск болезни | FN |
| Кредит | Отказ хорошему клиенту | Выдача кредита мошеннику | Зависит от суммы |

### Матрица стоимости (Cost Matrix)

| | Предсказано: + | Предсказано: - |
|---|---|---|
| **Реально: +** | 0 (или прибыль) | Cost(FN) |
| **Реально: -** | Cost(FP) | 0 |

**Общая стоимость:**

$$\text{Total Cost} = FP \cdot \text{Cost}(FP) + FN \cdot \text{Cost}(FN)$$

### Как учитывать стоимость

**Способ 1: Выбор порога**
- Выбрать порог, минимизирующий общую стоимость

**Способ 2: Взвешивание классов при обучении**
- Дать модели понять, что один класс важнее
- Больший вес для класса с более дорогими ошибками

**Способ 3: Cost-sensitive алгоритмы**
- Некоторые алгоритмы напрямую принимают матрицу стоимости

## Т8: Multi-class метрики

### One-vs-Rest и One-vs-One

Для многоклассовой классификации (больше 2 классов) используются стратегии сведения к бинарной:

**One-vs-Rest (OvR):**
- Отдельный классификатор для каждого класса против всех остальных
- Для K классов — K бинарных классификаторов

**One-vs-One (OvO):**
- Отдельный классификатор для каждой пары классов
- Для K классов — K(K-1)/2 бинарных классификаторов

### Averaging для многоклассовых метрик

Для метрик Precision, Recall, F1:

**Macro averaging (макро-усреднение):**

$$\text{Macro} = \frac{1}{K} \sum_{i=1}^{K} \text{Metric}_i$$

Среднее по всем классам. Все классы равны независимо от размера.

**Micro averaging (микро-усреднение):**

$$\text{Micro} = \frac{\sum TP_i}{\sum (TP_i + FP_i)}$$

Глобальный подсчёт по всем предсказаниям. Эквивалентно accuracy для сбалансированных данных.

**Weighted averaging (взвешенное усреднение):**

$$\text{Weighted} = \frac{\sum n_i \cdot \text{Metric}_i}{\sum n_i}$$

Среднее взвешенное по размеру классов. Учитывает дисбаланс.

### Когда использовать какой

| Метод | Когда использовать |
|-------|-------------------|
| Macro | Все классы важны одинаково, даже маленькие |
| Micro | Важна общая точность, классы сбалансированы |
| Weighted | Классы несбалансированы, важна общая точность |

**Пример:** Распознавание рукописных цифр (10 классов, примерно равные)
- Macro, Micro, Weighted дадут похожие результаты

**Пример:** Классификация типов мошенничества (5 классов, очень несбалансированных)
- Macro покажет, как модель работает на редких типах
- Weighted покажет общую эффективность

**Медицина (диагностика рака):**
- Главная метрика: Recall (пропустить рак нельзя)
- Дополнительная: Precision (но слишком много ложных тревог — перегрузка системы)
- Компромисс: Recall > 95%, F1 для баланса

**Спам-фильтр:**
- Главная метрика: Precision для класса «спам»
- Почему: отправить важное письмо в спам хуже, чем пропустить спам
- Цель: Precision > 99%

**Рекомендации видео:**
- Метрики: CTR, Time Watched, Diversity
- Почему: нужно не только кликнуть, но и досмотреть
- Скрытые метрики: Retention, Return Rate

**Кредитный скоринг:**
- Главная метрика: зависит от стратегии банка
- Консервативный банк: Precision (отказать плохим)
- Агрессивный банк: Recall (охватить всех хороших)
- Компромисс: максимизация прибыли (cost-sensitive)
